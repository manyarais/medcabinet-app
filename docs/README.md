# Pillio product docs

These docs help the team (and Cursor) raise product quality **without replacing the founder’s vision**.

They are based on `PRD.md`, then expanded with coaching questions and recommendations.  
**Final product decisions belong to the student product lead and team.**

## Start here next Cursor session (Manya)

1. Read **`MANYA_NEXT_SESSION.md`** — must-do handoff (low-risk OTC label search).
2. Read **`OTC_LABEL_SEARCH_SPEC.md`** — full UX + implementation spec.
3. Paste the build prompt from that brief (also in `PROMPT_PACK.md` #11).
4. Implement **behind the feature flag (default OFF)** so teachers can veto safely.

Dad checked in the docs only — **you implement the feature** when you’re next on.

## How to use these docs

1. Fill the open questions in `VISION.md` and `DEMO_SCRIPT.md` in your own words.
2. Before asking Cursor to build or polish, attach the relevant docs (see `PROMPT_PACK.md`).
3. Treat rules as **Must / Should / May** — not a creativity ban.
4. After each build session, update `CAPABILITIES.md` and score `AUDIT_SCORECARD.md`.

## Conflict guidance

- **Safety + PRD out-of-scope wins** over polish ideas (example: never recommend prescription meds for symptoms).
- **Her filled-in vision wins** over any recommendation in these docs.
- If docs disagree with code or `PRD.md` on scope, **ask the human** — do not invent features.
- Creativity is encouraged in tone, visuals, motion, and expression. New product scope needs a human yes.

## Spirit of this folder

These docs should make her think sharper, not smaller.

- Ask better questions when detail is missing.
- Recommend a direction when the PRD is silent.
- Always leave room for her to choose differently.
- Prefer evolving the hero experience over adding more pages.
